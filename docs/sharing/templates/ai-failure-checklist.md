> 在 Code Review 之前使用，定向检查 AI 生成代码的高风险区域。
> 不需要全量逐行 review——针对这六类模式做定向检查，覆盖率更高，成本更低。
>

> 1. 对照改动范围，判断哪几类适用（打 applicable 列）
> 2. 对适用的类别，逐条确认
> 3. 全部通过后再做常规 review

| 检查项 | 通过 |
|-------|------|

## 类别四：构建产物元数据

**适用场景**：改动涉及 Dockerfile、CI 配置、编译脚本、版本注入。

| 检查项 | 通过 |
|-------|------|
| `-ldflags -X` 的包路径是否正确？（Go：路径错误时静默跳过，不报错）| ☐ |
| 是否有 CI 步骤验证关键元数据已成功注入（版本号、构建时间）？| ☐ |
| 使用的基础镜像 tag 是否存在？（如 `golang:1.25-alpine` —— 1.25 版本不存在）| ☐ |
| 环境变量是否正确传入构建步骤？多层 Docker build 时是否有 ARG 声明？ | ☐ |
| Release 产物是否有自动化测试验证"功能符合预期"（而不只是"能运行"）？ | ☐ |

**CI 验证参考**：
```bash
# 在 CI 里加这个断言，防止版本注入静默失败
```

## 类别五：数据库业务语义

**适用场景**：改动涉及数据库 Schema、upsert、批量同步、数据迁移。

| 检查项 | 通过 |
|-------|------|
| 每张表的"业务唯一键"是否和"数据库主键"区分清楚了？| ☐ |
| upsert 的 `ON CONFLICT` 子句用的是哪个字段？是业务唯一键，还是内部 UUID？| ☐ |
| 多节点场景下，同一业务实体在不同节点是否会生成不同的内部 ID？| ☐ |
| 数据迁移/同步时，幂等性如何保证？`INSERT OR IGNORE` 还是 `ON CONFLICT DO UPDATE`？| ☐ |
| Schema 里的 `UNIQUE` 约束是否覆盖了所有业务唯一性要求？| ☐ |
| 新增字段是否有合理的默认值？迁移时旧记录会如何处理？| ☐ |

**常见错误示例**：
```sql
-- ❌ 错误：用内部 UUID 作为 upsert 冲突键
INSERT INTO targets (id, url, ...) VALUES (...)
ON CONFLICT(id) DO UPDATE SET ...;
-- 多节点场景下同一 URL 的 id 不同，冲突永不命中

-- ✅ 正确：用业务唯一键
INSERT INTO targets (id, url, ...) VALUES (...)
ON CONFLICT(url) DO UPDATE SET id=excluded.id, ...;
```

## 类别六：前端视觉交互

**适用场景**：改动涉及 HTML/CSS/JS、图表、滚动、响应式布局、动态渲染。

| 检查项 | 通过 |
|-------|------|
| 图表（Chart.js/ECharts 等）是否有固定高度的父容器？（responsive + no maintainAspectRatio 会无限撑大）| ☐ |
| 是否在浏览器里实际打开并滚动/缩放验证过？ | ☐ |
| 动态加载的内容（AJAX/SSE）是否有加载失败的 UI 状态？ | ☐ |
| 长文本/长 URL 是否会撑破布局？ | ☐ |
| 表格/列表数据量很大时是否有分页或虚拟滚动？ | ☐ |
| 深色模式/浏览器缩放是否会破坏布局？（如有要求）| ☐ |

**Chart.js 常见修复参考**：
```html
<!-- ❌ 错误：无固定高度父容器，会无限增高 -->
<canvas id="myChart"></canvas>

<!-- ✅ 正确：用固定高度的 div 包裹 -->
<div style="height: 300px; position: relative;">
    <canvas id="myChart"></canvas>
</div>
```
