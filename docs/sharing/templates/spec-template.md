# Spec 设计规格模板

> 在给 AI 分配实现任务之前，先用此模板写 Spec，让 AI 帮你完成初稿，然后你来 review。
> 删除所有以「> 」开头的注释行后使用。
> 核心原则：Spec 是对齐文档，不是需求文档。它回答"做什么"和"为什么这样做"，不回答"怎么做"。

---

# Spec：[功能名称]

**日期**: YYYY-MM-DD
**目标版本**: vX.Y.Z
**状态**: Draft / In Review / Approved
**作者**: [你的名字 或 Claude Model Name]

---

## 问题描述

> 必须用具体反例或具体场景，不用抽象描述。
> 好的问题描述让读者（包括 AI）能独立判断"解决方案 A 是否解决了这个问题"。

### 现状

[描述现在的行为]

### 问题 / 缺失

[用具体数据、日志、操作步骤描述问题，或用场景描述能力缺失]

### 具体案例（必填）

> 把问题用最具体的形式写出来。能用代码/日志/操作步骤描述的，不用自然语言描述。

```
[粘贴实际的错误日志、操作步骤、或具体的碰撞/冲突场景]

示例：
用户 alice123 生成 Key → 字符集 {a,l,i,c,e,1,2,3}
用户 321ecila 生成 Key → 字符集 {3,2,1,e,c,i,l,a}（完全相同）
攻击者可以枚举出同时通过两个用户验证的 Key。
```

## 完成标准（Success Criteria）

> 这是 Spec 最重要的部分。AI 会对照这个列表验收自己的实现。
> 每一条必须是可以客观判断"是否满足"的，避免模糊表述。

### 功能性

- [ ] [功能要求 1，例如"用户名 A + secret → 永远产生同一个 Key（幂等性）"]
- [ ] [功能要求 2，例如"旧格式 Key 在升级后返回 401，不返回 400"]
- [ ] [功能要求 3，例如"keygen_secret < 32 字符时，启动时打印错误并退出，退出码非 0"]

### 性能

- [ ] [性能要求 1，例如"Key 生成耗时 < 1ms（单次调用，不含网络）"]
- [ ] [性能要求 2，例如"Key 验证（无缓存，1000 用户线性扫描）< 10ms"]

### 测试

- [ ] [测试要求 1，例如"generator.go 覆盖率 100%"]
- [ ] [测试要求 2，例如"10,000 个随机用户名，无两个用户生成同一 Key"]
- [ ] [测试要求 3，例如"旧格式 Key 被正确拒绝，有对应的测试用例"]

### 文档

- [ ] [文档要求 1，例如"UPGRADE.md 标注 Breaking Change 及升级步骤"]
- [ ] [文档要求 2，例如"用户文档更新：说明'重新生成'的幂等性行为"]
- [ ] [文档要求 3，例如"运维文档更新：keygen_secret 轮换的影响"]

---

## Out of Scope（本次不做的）

> 明确写出"看起来相关但本次不做"的事情。
> 这防止 AI 自发扩展范围，也避免 reviewer 提出"为什么没做 X"的问题。

本次 Spec **不包含**以下内容：

- [不做的事 1，例如"双 secret 支持（新旧 Key 并存过渡期）—— 列入 Roadmap v3.x"]
- [不做的事 2，例如"用户级别的 Key 独立过期时间"]
- [不做的事 3，例如"前端 UI 的视觉改版"]

---

## 实现检查清单（Implementation Checklist）

> 这是 Spec 和 Plan 之间的桥梁。
> Spec 说"做什么"，Checklist 说"哪些文件/步骤需要改"，Plan 说"怎么改"。
> AI 实现完成后对照此清单逐项确认。

- [ ] 更新配置结构体（新增必填字段）
- [ ] 更新配置校验函数
- [ ] 实现核心算法（generator.go）
- [ ] 实现验证逻辑（validator.go）
- [ ] 更新调用链（所有传入新参数的地方）
- [ ] 补充单元测试
- [ ] 补充 E2E 测试（如果有集成路径）
- [ ] 更新 UPGRADE.md（Breaking Change 说明）
- [ ] 更新 manual.md / docs（用户可见的行为变化）
- [ ] 运行全量测试套件，0 失败

---

> ## Spec Review 检查清单（给 reviewer 用）
>
> 在批准 Spec 进入实现阶段前，确认：
>
> - [ ] 问题描述用了具体反例，不是抽象描述
> - [ ] 每个设计决策都有"已拒绝的备选方案"
> - [ ] Success Criteria 每条都是客观可判断的（不是"功能完整"这种模糊表述）
> - [ ] 用户预期变化（包括 UX 层面）已分析
> - [ ] Out of Scope 涵盖了所有"容易误解为应该做"的事情
> - [ ] Breaking Changes 已明确标注
